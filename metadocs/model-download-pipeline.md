# Загрузка и проверка локальной модели

Эта заметка описывает не UI, а техническое ядро скачивания модели: как приложение ставит загрузку в системный `DownloadManager`, где хранит файл, как переживает перезапуск экрана/процесса, почему нельзя доверять одному факту существования файла и как сейчас устроена проверка модели через размер и SHA256.

## Главная идея

Модель нельзя считать установленной только потому, что в папке появился файл.

Для `.litertlm`-модели нормальное состояние такое:

1. загрузка была поставлена в системный `DownloadManager`;
2. Android вернул `downloadId`;
3. `downloadId` сохранен в `SharedPreferences`;
4. после `DownloadManager.STATUS_SUCCESSFUL` файл проверен по ожидаемому размеру;
5. файл проверен по ожидаемому SHA256;
6. только после этого приложение считает модель установленной.

То есть финальный флаг "модель скачана" должен быть производным от проверки файла, а не отдельной истиной сам по себе.

## Основные классы

### `KsetaDownloader`

`KsetaDownloader` - минимальный контракт загрузчика:

```kotlin
fun interface KsetaDownloader {
    fun download(url: String, fileName: String, modelName: String): Long
}
```

Он отвечает только за постановку загрузки и возвращает `Long` id задачи. Этот id потом нужен, чтобы спрашивать Android о состоянии загрузки.

### `KsetaModelDownloader`

`KsetaModelDownloader` - низкоуровневая обертка над Android `DownloadManager`.

Он делает несколько вещей:

- получает системный сервис `Context.DOWNLOAD_SERVICE`;
- перед новой загрузкой удаляет старый файл модели;
- создает `DownloadManager.Request`;
- задает URL, title, description, network flags и destination;
- ставит задачу через `enqueue`;
- умеет проверить наличие файла;
- умеет проверить валидность файла через размер и SHA256;
- умеет удалить файл модели.

Важная деталь: `download()` сначала вызывает `deleteModelFile(modelName, fileName)`.

Это нужно из-за поведения отмененных или оборванных загрузок: в папке модели может остаться пустой или частичный файл. Если его не удалить перед новой загрузкой, приложение рискует работать с мусорным артефактом или получить конфликт записи.

### `KsetaDownloadWrapper`

`KsetaDownloadWrapper` - модельно-специфичная обертка над общим downloader.

Именно здесь сейчас лежит "паспорт" Gemma:

```kotlin
const val GEMMA_4_E2B_MODEL_NAME = "gemma-4-e2b"
const val GEMMA_4_E2B_URL =
    "https://huggingface.co/litert-community/" +
        "gemma-4-E2B-it-litert-lm/resolve/main/gemma-4-E2B-it.litertlm"
const val GEMMA_4_E2B_FILE_NAME = "gemma-4-e2b-it.litertlm"
const val GEMMA_4_E2B_SIZE_BYTES = 2_588_147_712L
const val GEMMA_4_E2B_SHA256 =
    "181938105e0eefd105961417e8da75903eacda102c4fce9ce90f50b97139a63c"
```

Это нормальное место для открытых метаданных модели: URL, имя файла, ожидаемый размер и публичный HF/LFS SHA256 не являются секретами.

## Где хранится модель

Сейчас используется:

```kotlin
setDestinationInExternalFilesDir(
    context,
    null,
    "models/$modelName/$fileName"
)
```

Физически это app-specific external files storage:

```text
Android/data/<package>/files/models/gemma-4-e2b/gemma-4-e2b-it.litertlm
```

Рядом с файлом модели теперь выделяется отдельная директория runtime-кэша:

```text
Android/data/<package>/files/models/gemma-4-e2b/cashed
```

Именно этот путь передается в LiteRT-LM как `EngineConfig.cacheDir`, чтобы временные
артефакты общения с моделью лежали рядом с конкретной моделью и не смешивались с
другими файлами приложения.

Плюсы такого пути:

- приложению не нужен ручной выбор директории;
- это не общий `Downloads`, а пространство приложения;
- Android сам понимает, куда писать файл;
- путь можно потом передать в LiteRT-LM как локальный файл модели.

Минус: если пользователь удалит данные приложения или само приложение, этот файл может быть удален вместе с app-specific storage.

## Что значит `downloadId`

`downloadId` - это не id модели. Это id конкретной задачи в системном `DownloadManager`.

Он нужен для запросов вида:

```kotlin
DownloadManager.Query().setFilterById(downloadId)
```

По нему приложение узнает:

- задача еще существует или уже пропала;
- сколько байт скачано;
- какой общий размер известен Android;
- текущий статус: running, paused, successful, failed.

В коде проекта `downloadId` сохраняется в `SharedPreferences`, чтобы после пересоздания `Activity` или возвращения в приложение можно было снова привязаться к той же системной загрузке.

## Почему нужен `SharedPreferences`

`remember` и `rememberSaveable` живут в Compose/UI-жизни. Они не являются надежным хранилищем для фоновой системной загрузки.

`SharedPreferences` здесь хранит только один факт:

```text
active_model_download_id -> Long
```

Сценарий:

1. пользователь нажал скачать;
2. `DownloadManager.enqueue(...)` вернул id;
3. id записали в `SharedPreferences`;
4. приложение свернули или экран пересоздался;
5. при новом входе приложение читает сохраненный id;
6. если id не равен `NO_DOWNLOAD_ID`, оно опрашивает `DownloadManager`.

Если загрузка завершилась, упала, отменена или запись в `DownloadManager` пропала, id удаляется из `SharedPreferences`.

## Что происходит при отмене

Отмена делается через:

```kotlin
downloadManager.remove(downloadId)
```

Это просит системный `DownloadManager` убрать задачу. После этого проект дополнительно:

- сбрасывает локальный `currentDownloadId`;
- чистит `SharedPreferences`;
- вызывает `deleteGemma4E2BFile()`.

Удаление файла после отмены важно. Иначе может остаться пустышка или частично скачанный `.litertlm`, который внешне похож на модель, но не является рабочей моделью.

## Что происходит при перезапуске приложения

При старте экрана приложение читает `downloadId` из `SharedPreferences`.

Если `downloadId` активный, приложение опрашивает системный `DownloadManager`.

Если `downloadId` отсутствует, но в папке есть кандидат-файл модели, приложение запускает проверку:

1. файл существует;
2. файл не пустой;
3. размер совпадает с `GEMMA_4_E2B_SIZE_BYTES`;
4. SHA256 совпадает с `GEMMA_4_E2B_SHA256`.

Если проверка прошла, модель считается установленной.

Если проверка не прошла, файл удаляется, а модель считается не установленной.

## Что значит `STATUS_SUCCESSFUL`

`DownloadManager.STATUS_SUCCESSFUL` означает только то, что Android считает сетевую загрузку завершенной.

Это не означает автоматически:

- что файл является нужной моделью;
- что файл не битый;
- что HF не отдал другую версию;
- что файл можно открыть через LiteRT-LM.

Поэтому после `STATUS_SUCCESSFUL` код делает отдельную integrity-проверку через `hasValidGemma4E2BFile()`.

## Проверка файла модели

Сейчас есть два уровня проверки.

### Быстрая проверка-кандидат

```kotlin
modelFile.exists() && modelFile.length() > 0L
```

Она отвечает только на вопрос: "есть ли вообще что-то похожее на файл модели?"

Эта проверка нужна, чтобы не показывать процесс индексации на чистом устройстве, где файла нет.

### Полная проверка

```kotlin
modelFile.exists()
modelFile.isFile
modelFile.length() == expectedSizeBytes
sha256(file) == expectedSha256
```

Только эта проверка может дать право сказать: "да, локальный файл совпадает с ожидаемым HF-артефактом".

## Откуда берется SHA256

SHA256 берется не с телефона после скачивания как "истина". Иначе приложение просто доверяло бы любому скачанному мусору.

Ожидаемый SHA256 берется из публичных метаданных HuggingFace/LFS/Xet для конкретного файла. Потом приложение локально считает SHA256 скачанного файла и сравнивает:

```text
actualSha256 == expectedSha256
```

Так приложение доказывает не просто "файл есть", а "файл совпадает с известным артефактом модели".

## Почему SHA считается на `Dispatchers.IO`

Файл модели весит примерно 2.59 GB. Подсчет SHA256 читает весь файл от начала до конца.

Такую операцию нельзя выполнять на UI-потоке. Поэтому проверка вызывается через:

```kotlin
withContext(Dispatchers.IO) {
    wrapper.hasValidGemma4E2BFile()
}
```

Это не делает проверку мгновенной, но не блокирует основной поток интерфейса.

## Почему URL на `main` - технический риск

Сейчас URL указывает на:

```text
.../resolve/main/gemma-4-E2B-it.litertlm
```

Это удобно на этапе разработки, но `main` может измениться. Если HuggingFace обновит файл, старый SHA256 перестанет совпадать.

Для более строгой версии лучше закрепить URL на конкретный revision/commit:

```text
.../resolve/<commit-sha>/gemma-4-E2B-it.litertlm
```

Тогда URL, размер и SHA256 будут описывать один и тот же неизменяемый артефакт.

## Что будет, если приложение убрать из списка последних

Обычный `DownloadManager` - системная очередь загрузок. Если приложение просто свернули или убрали Activity из recents, системная загрузка может продолжиться.

Но есть важные ограничения Android:

- если пользователь делает force stop приложения, поведение может быть жестче;
- если система очищает задачу или запись `DownloadManager`, приложение при следующем входе увидит, что id больше не найден;
- если файл остался частичным, проверка размера/SHA его забракует и удалит.

Именно поэтому состояние нельзя держать только в UI и нельзя верить только файлу на диске.

## Что такое "индексация" в текущем коде

В текущей версии "индексация модели" - это пользовательское название фазы проверки файла.

Технически сейчас не строится embedding-index и не создается поисковый индекс. Происходит integrity check:

- проверка кандидата;
- проверка размера;
- подсчет SHA256;
- перевод модели в состояние installed или broken.

Когда позже появится LiteRT-LM, сюда логично добавить еще один этап: пробную инициализацию runtime на этом файле. Тогда модель будет считаться не только скачанной и целой, но и реально открываемой движком.

## Как это связано с LiteRT-LM

Downloader не должен запускать модель.

Его задача заканчивается на результате:

```text
valid local model file path
```

Следующий слой должен быть отдельным runtime-классом, например:

```text
KsetaLiteRtModelRuntime
```

Он уже получает путь к `.litertlm`, создает LiteRT-LM engine, инициализирует модель и выполняет генерацию.

Так разделение остается чистым:

- `KsetaModelDownloader` доставляет и проверяет файл;
- `KsetaDownloadWrapper` хранит метаданные конкретной модели;
- `MainActivity` сейчас временно связывает состояние загрузки с экраном;
- будущий LiteRT-LM runtime отвечает за инференс.

## Что стоит вынести следующим шагом

Сейчас часть orchestration-логики живет в `MainActivity`: чтение `SharedPreferences`, опрос `DownloadManager`, обработка статусов, запуск проверки.

Для следующей версии это лучше вынести из `Activity` в отдельный не-UI класс, например:

```text
KsetaModelInstallManager
```

Его ответственность:

- старт загрузки;
- отмена загрузки;
- восстановление активной загрузки;
- проверка файла;
- удаление битого файла;
- выдача состояния модели.

Тогда UI будет только подписываться на состояние:

```text
NOT_INSTALLED
DOWNLOADING
VERIFYING
INSTALLED
FAILED
CANCELLED
BROKEN
```

Это будет чище, чем хранить много флагов отдельно.

## Главные правила этого механизма

1. Не доверять `exists()` как признаку установленной модели.
2. Не доверять `STATUS_SUCCESSFUL` как признаку рабочей модели.
3. Перед новой загрузкой удалять старый файл.
4. После отмены удалять частичный файл.
5. После ошибки удалять частичный файл.
6. `downloadId` хранить отдельно от UI-состояния.
7. SHA256 считать в фоне.
8. Метаданные модели хранить рядом с модельным wrapper.
9. Для релиза закрепить URL на конкретный HF revision.
10. Инференс через LiteRT-LM делать отдельным слоем, не в downloader.
